# E2E（Playwright）規約

- 要素の特定はユーザーに見える情報（ロール + アクセシブルネーム、表示テキスト、ラベル）で行い、`data-testid` などのテスト専用属性で特定しない
  - 理由: `data-testid` は本番 UI に現れずユーザー体験を反映しないため、その属性が壊れても実害のある UI 崩れを検知できない。ロールやアクセシブルネームで特定すれば、E2E が実際のユーザー操作・アクセシビリティの担保も兼ねられる
  - 指針: `getByRole` / `getByText` / `getByLabel` を優先する。一意に絞り込めない場合は、要素に付ける `aria-label` 等をユーザーにとって意味のある名前にして特定する（例: 記事カードの開くトリガに `記事「<タイトル>」を開く`）
  - 注意: 特定のためだけに `data-*` 属性を足さない。名前が衝突する場合は、テスト都合ではなくアクセシビリティ上も妥当な命名で解消する

- シナリオテストのファイルはデバイス種別ではなくシナリオ単位で構成し、`*.mobile.test.ts` のようなデバイス別ファイルは作らない
  - 理由: 直列実行のため観点はシナリオで束ねる方が読み取りやすく、同一シナリオのデスクトップ・モバイルが別ファイルに散ると対応関係を追いづらい
  - 指針: 1つのシナリオファイル内でデスクトップ・モバイル両方のケースを扱う。同一手順を複数 viewport で検証する場合は viewport の表（`Object.entries` で `describe` を回す等）で並べ、テスト本体の重複を避ける

- `scenario/` にはユーザーの一連の流れ（フロー）を表すテストだけを置き、単体の UI 挙動確認は置かない
  - 理由: 「シナリオ」の粒度が曖昧になり、ファイルが何を保証しているのか読み取りづらくなる
  - 指針: 特定コンポーネントの開閉・トグルなど流れを伴わない挙動確認は、シナリオの一部として組み込むか、シナリオ以外の適切な場所で扱う
