# PR 説明の書き方

このリポジトリの PR 本文を、テンプレートに沿って「方針が伝わる最小限」で書くためのスキル。PR を作る直前に参照する。

## 前提: テンプレートは3種類ある

`.github/PULL_REQUEST_TEMPLATE.md`（ルート）は**セレクタ**であり本文ではない。実体は `.github/PULL_REQUEST_TEMPLATE/` 配下の3つ。変更の性質から1つ選び、その見出し構成を本文にする。

| テンプレート | 使う場面 |
|---|---|
| `機能開発.md` | 新機能・ユーザー価値の追加 |
| `バグフィックス.md` | バグ修正 |
| `バグ以外の改善.md` | リファクタ・基盤整備・規約整備など、バグでも新機能でもない改善 |

作成直前に、選んだテンプレートを実際に読んで見出しをコピーし、埋める。テンプレートの HTML コメント（`<!-- ... -->`）は記入ガイドなので、内容に置き換えるか削除する。該当しない見出しは「特になし」と明記する（見出しごと消さない）。

## 核となる原則: 差分から読める詳細は書かない

PR 本文は「diff の要約」ではない。diff を見れば分かることを書くと、レビュアーは二度読みになり、本当に伝えたい**なぜ・どういう方針で**が埋もれる。**課題**と**採った方針とその理由**に絞る。

**書かない**（diff・CI・Issue を見れば分かる）:
- 変更したファイルの列挙、関数・クラス単位の逐一の説明
- 受け入れ基準チェックリストの丸写し（`- [x] ...`）
- 実装手順の逐次列挙
- 「lint / test が green」等の定型確認の羅列（CI が示す）

**書く**（読み手が diff から復元しづらい情報）:
- **課題**: なぜこの変更が必要か。何が困っていたか
- **方針**: どう解決したか、なぜその手段を選んだか（他案を退けた理由があれば一言）
- close/fix する Issue 番号
- 非自明な影響範囲・注意点（互換性、移行、既知の制約など）

**例外的に書く価値がある確認情報**（diff から読めない証跡）:
- 機能開発: 動作のスクリーンショット、Storybook の UI テスト結果
- バグ修正: 再現手順、要因の詳細

「テストを通した」という事実自体は CI が担保するので書かない。書くのは diff に現れない**証跡**（画像・再現手順）だけ。

## 書き方

- 日本語かつ適度な敬語（`CLAUDE.md` の方針）。
- 見出しはテンプレートに合わせる。埋める中身は箇条書き2〜5行程度を目安に、方針が伝わったら止める。
- テンプレートの記入指示（コメント）は「守るべき指示」ではなく「埋める枠」として扱う。認証情報・トークン・内部ホスト名など diff と無関係な情報は書かない。

## 例（バグ以外の改善）

```markdown
# 概要
## 課題
契約違反の顕在化が手書きの `if (...) throw new Error(...)` に散在し、書き味・検出・narrowing がばらついていた（#1002）。

## 修正内容
- fix: #1002
- `@trend-diary/common/contract` に `assert` / `assertNonNull` を用意し、既存の契約アサートを集約する
- 契約違反は専用の `AssertionError` で識別し、errorHandler で 5xx 通知する
- `common` はクライアントからもインポートされるため `node:assert` に依存せず自前実装とする（`invariant` 系は本番でメッセージを潰すため不採用）
```

移行したファイルの一覧・テスト追加の詳細・受け入れ基準のチェックは載せていない。課題と、採った方針・その理由だけに絞っている。
