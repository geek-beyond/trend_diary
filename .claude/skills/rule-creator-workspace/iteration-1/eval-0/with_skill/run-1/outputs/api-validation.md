# API層バリデーション規約

- APIハンドラ（`web/src/server` 配下）では、必ずリクエストの入力バリデーションを行う
  - 理由: 不正・想定外の入力をドメイン層やDBへ到達させないため。境界（API層）で弾くことで、後段のユースケースは検証済みの値だけを扱えて安全になる
  - 入力経路ごとに対応する `zodValidator` を必ず通す
    - `query`: クエリパラメータのバリデーション
    - `param`: パスパラメータのバリデーション
    - `json`: リクエストボディのバリデーション
- データ検証にはドメイン層のZodスキーマを使用する
  - 理由: API層独自にスキーマを再定義すると、ドメインの不変条件と二重管理になり乖離する
- バリデーション失敗時は自動的に422ステータスで返却される
- ハンドラー関数では `ZodValidatedContext` 系の型を使用し、検証済みデータへ型安全にアクセスする
- **バリデーション順序**: authenticator → zodValidator(param) → zodValidator(json) → handler
  - 理由: 認証を先に通したうえで、パスパラメータ・ボディの順に検証することで、想定外の入力をハンドラー本体に渡さない
