# Release

リリースのたびに必要な一連の手順を、順番どおりに実行するスキル。

逐次手順かつ「呼び出したときだけ動く」ワークフローなので、ルール（`.claude/rules/`）ではなくスキルとして定義する。各ステップは前のステップの成功を前提に進める。途中で失敗したら止めて原因を報告し、未完了のステップ（特にタグ・デプロイ）を勝手に巻き戻したり再実行したりしない。

## 前提・確認

- リリース対象のブランチ・コミットが正しいか確認する（通常はデフォルトブランチの最新）
- 作業ツリーがクリーンか確認する（未コミットの変更がないこと）
- リリースするバージョン番号を確定する（引数で渡されていなければユーザーに確認する）

## 手順

### 1. CHANGELOG を更新する

- リポジトリの CHANGELOG（無ければ作成方針をユーザーに確認）に、今回のリリース分のエントリを追記する
- バージョン・日付・主な変更点を記載する
- 変更をコミットする（コミットメッセージ規約があれば従う）

### 2. バージョンタグを打つ

- 確定したバージョンで注釈付きタグを作成する
  - 例: `git tag -a v1.2.0 -m "v1.2.0"`
- タグをリモートへ push する
  - 例: `git push origin v1.2.0`

### 3. wrangler deploy する

- デプロイ対象のパッケージを確認する（このリポジトリでは `application/packages/web` と `application/packages/cron` に `wrangler.toml` がある）
- 対象ごとに `wrangler deploy` を実行する
- デプロイ結果（成功・デプロイ先・バージョン）を確認する

### 4. Discord に完了通知を送る

- リリース完了を Discord に通知する（Webhook URL は環境変数・シークレットから取得し、ハードコードしない）
- 通知内容にはバージョンとデプロイ結果の要点を含める

## 完了報告

- 実行したステップ、打ったタグ、デプロイ先、通知の送信可否をまとめてユーザーに報告する

## やってはいけないこと

- ステップを飛ばす・順序を入れ替える（CHANGELOG とタグはデプロイより前）
- 失敗したのに次のステップへ進む
- Discord Webhook 等のシークレットをコード・ログに残す
