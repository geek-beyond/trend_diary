Status: Accepted

Relevant PR:

# Context

これまでアプリは単一パッケージ（`application/`）で構成され、web worker と cron worker という 2 つのデプロイ成果物が、同じ `package.json`（React 等のフロント依存を含む約 40 依存）からビルドされていた。

このため以下の課題があった。

- **cron の不要な肥大化**: cron は `rss-parser` / `drizzle` / domain / datastore / common しか使わないのに、フロント依存ごとビルドされバンドルが膨らみ、コールドスタートにも不利だった。
- **レイヤー境界が規約頼み**: レイヤー間の依存方向は規約でしか守られておらず、Biome の `noRestrictedImports` も off だったため物理的な強制力がなかった。
- **通知ロジックの重複**: Discord 通知の送信ロジックが web と cron で重複していた。

これを解消するため、pnpm workspace を活用してレイヤー / デプロイ単位でパッケージを分離した。本 ADR は、その分離プロジェクト（親イシュー #685、サブイシュー #677〜#684）で確定した設計判断を記録する。

# Decision

`application/packages/*` 配下に、以下のパッケージ構成（pnpm workspace）で分離する。

```
@trend-diary/common       … 葉（最下層の共有基盤）
@trend-diary/datastore    … infrastructure/{drizzle-orm, rdb}
@trend-diary/notification … 通知基盤（web/cron 共有・1 パッケージ）
@trend-diary/domain       … → common, datastore
@trend-diary/cron         … → common, datastore, domain, notification（React 非依存・別 worker）
@trend-diary/web          … 上記全部 ＋ api.ts ＋ ChatNotifier アダプタ
@trend-diary/lint         … dev tooling（Biome 設定・プラグイン）
@trend-diary/e2e          … E2E テスト
```

確定した個別の設計判断は以下のとおり。

1. **`infrastructure` は DDD 最上位層ではなく「主にデータストア」を指す**
   ここで言う `infrastructure`（= `datastore` パッケージ）は、DDD の最上位インフラ層ではなく、`common` と同様の下位共有基盤である。`drizzle-orm` / `rdb` を内包する。

2. **依存方向は `common ← datastore ← domain ← {web, cron}` のクリーンな DAG**
   `domain → datastore` は**正方向であり違反ではない**。domain がデータストア基盤に依存するのは想定どおりの向き。

3. **`api.ts`（`hc<typeof app>` の RPC 型ブリッジ）は web 内部に閉じる**
   実利用は web/client のみのため、共有基盤には出さず web パッケージ内部に閉じる。

4. **`ChatNotifier` ポートは web 所有のまま**
   `ChatNotifier` は web の 5xx エラー報告用ポートであり、web 所有のままで正しい。通知パッケージへは移動しない。

5. **通知基盤は 1 パッケージとしてまとめて切り出す**
   通知基盤（Discord 通知）は web のエラー通知と cron のジョブ失敗通知の双方で使う共有基盤である。web 側などへ切り崩して分散させず、`@trend-diary/notification` として 1 パッケージにまとめる。`ChatNotifier` インターフェースは web に残しつつ、通知パッケージが web に依存しない形にする。

6. **client / server のパッケージ分割はしない**
   client↔server は Hono RPC（`hc<typeof app>`）で型結合しているため、分割すると型の往復が壊れる。両者は web パッケージ内に同居させる。

7. **パッケージ切り出し時は CI も毎回分離する**
   パッケージを切り出す各イシューでは、そのパッケージ専用の CI（テスト / lint / 型チェック等）も必ず分離して追加する。CI 全体構成の詳細は [CI / CD 全体構成](/docs/how_to_guides/ci_cd_overview.md) を参照。

## Reason

- **cron の依存スリム化が最大の目的**: cron を React 非依存の独立パッケージにすることで、バンドルサイズとコールドスタートを改善できる。
- **境界の物理強制**: レイヤー依存を「規約」ではなく「package 境界」で保証する。上位を import しようとすると依存解決の時点で破綻するため、違反が物理的に起きにくくなる。
- **重複解消**: 通知基盤を 1 パッケージにまとめることで、web / cron 双方が同じ実装を参照し重複が消える。
- **共有基盤は分散させない**: 通知基盤を web 等へ切り崩すと「どこに通知ロジックがあるか」が曖昧になり、再び重複や責務の分散を招くため、1 パッケージとして保つ。
- **RPC 型結合の尊重**: client/server を無理に割ると Hono RPC の型推論が壊れ、型安全な API 呼び出しという最大の利点を失うため分割しない。

検討した他の選択肢としては「通知ロジックを web へ寄せて cron から参照する」案があったが、cron を web に依存させると cron のフロント依存スリム化という最大の目的に反するため却下した。

# Consequences

- cron がフロント依存から切り離され、cron worker のバンドルが軽量化された（dry-run でも web 約 4.4MB に対し cron 約 0.6MB）。
- レイヤー境界が package 境界として物理的に強制されるようになった。
- Discord 通知ロジックの重複が解消された。
- パッケージごとに `package.json` / CI が分かれ、変更パッケージのみをテストできるようになった（CI の変更検知については別ドキュメント参照）。
- 一方で、パッケージを 1 つ増やすたびに `pnpm-workspace.yaml`・`package.json`・専用 CI を整える手間が増える。これは「CI も毎回分離する」ルールとして明文化することで運用負荷の予見可能性を担保した。
