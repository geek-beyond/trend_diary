# 認証フローのエントリポイントをHono APIに統一する

Status: Accepted

Relevant PR:

https://github.com/geek-beyond/trend_diary/pull/788

# Context

login/signup/セッション確認は React Router の action/loader からドメインユースケース（`createAuthUseCase`）を直接呼び出す実装になっていた。

一方で Hono API 側には `/api/auth/signup`・`/api/auth/login` が存在し、rateLimiter（`AUTH_RATE_LIMITER` binding, 10req/60s）・CSRF・zodValidator を適用していたが、これらのエンドポイントは本番コードから呼ばれておらずテスト専用になっていた。

その結果、以下の問題が生じていた。

- ブルートフォース対策が Turnstile CAPTCHA のみに依存し、レート制限が実効性を持っていなかった
- 認証機能内でも logout/me は API（Hono RPC）経由、login/signup は action 直呼びと経路が分裂していた

## References

# Decision

認証のエントリポイントを Hono API（`/api/auth/*`）に一本化し、クライアントは logout・記事系と同じく**ブラウザから直接 Hono RPC を呼ぶ**。

- login/signup はクライアントサイドのフック（`use-login` / `use-signup`）がフォーム値の検証と RPC 呼び出しを行う。`Set-Cookie` の保存・送信はブラウザの標準動作に任せる
- セッション確認は `useSession`（SWR + `GET /api/auth/me`）で行う
- client 配下からのドメインユースケース生成は lint で禁止し、型・定数・スキーマの import のみ許可する。サーバー専用パッケージ（datastore / notification）は import 自体を禁止する

## Reason

- レート制限・CSRF・バリデーションといった横断的関心事が Hono ミドルウェアとして既に実装済みであり、API 経路に一本化すればそれらを実際の認証経路へそのまま適用できるため
- 検討した代替案
  - 案B: action 経路を正とし API 側の signup/login を削除し、action 側にレート制限を再実装する → Hono ミドルウェア資産の再実装になり二重管理が解消しないため不採用
  - 案C: action/loader を薄いBFFとし、Hono app を in-process（`app.request`）で呼び出す → Cookie・`Set-Cookie`・各種ヘッダーの手動転送が必要になる上、フォーム由来リクエストの JSON 変換が `hono/csrf` の検査対象（フォーム系 Content-Type）を外れるため CSRF 対策の追加実装も必要になる。複雑性が利点（form/action による progressive enhancement）に見合わないため不採用

# Consequences

- レート制限・バリデーションが実際のログイン/サインアップ経路に適用されるようになった
- CSRF は既存の仕組みのままで成立する（フォーム送信が廃止され、JSON リクエストは「HTMLフォームから送信できない」という `hono/csrf` の前提が保たれるため）
- Cookie の往復がブラウザの標準動作で完結し、BFF的な転送機構が不要になった
- React Router の action/loader から認証処理が消え、JS 無効環境でのフォーム送信（progressive enhancement）は非対応になった
- ログイン状態の判定がクライアントサイド（SWR）になり、SSR 直後は未ログイン表示で描画され、セッション確認後に更新される
