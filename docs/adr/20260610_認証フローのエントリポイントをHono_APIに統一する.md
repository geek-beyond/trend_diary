# 認証フローのエントリポイントをHono APIに統一する

Status: Accepted

Relevant PR:

# Context

login/signup/セッション確認は React Router の action/loader からドメインユースケース（`createAuthUseCase`）を直接呼び出す実装になっていた。

一方で Hono API 側には `/api/auth/signup`・`/api/auth/login` が存在し、rateLimiter（`AUTH_RATE_LIMITER` binding, 10req/60s）・CSRF・zodValidator を適用していたが、これらのエンドポイントは本番コードから呼ばれておらずテスト専用になっていた。

その結果、以下の問題が生じていた。

- ブルートフォース対策が Turnstile CAPTCHA のみに依存し、レート制限が実効性を持っていなかった
- 認証機能内でも logout/me は API（Hono RPC）経由、login/signup は action 直呼びと経路が分裂していた

## References

# Decision

認証のエントリポイントを Hono API（`/api/auth/*`）に一本化する。

- React Router の action/loader は薄いBFFコントローラとし、Hono app を in-process（`app.request`）で呼び出す。`Set-Cookie` は API レスポンスから action のレスポンスへ転送する
- CSRF 対策は in-process 呼び出しの入口（auth-api.ts の `postAuthApi`）で同一オリジン検証として実装する。フォーム由来のリクエストを JSON へ変換すると `hono/csrf` の検査対象（フォーム系 Content-Type）から外れるため、API 側の csrf ミドルウェアには委譲できない
- client 配下からのドメインユースケース生成は lint で禁止し、型・定数・スキーマの import のみ許可する。サーバー専用パッケージ（datastore / notification）は import 自体を禁止する

## Reason

- レート制限・CSRF・バリデーションといった横断的関心事が Hono ミドルウェアとして既に実装済みであり、API 経路に一本化すればそれらを実際の認証経路へそのまま適用できるため
- 検討した代替案
  - 案B: action 経路を正とし API 側の signup/login を削除し、action 側にレート制限を再実装する → Hono ミドルウェア資産の再実装になり二重管理が解消しないため不採用
  - ブラウザから直接 RPC（`client.auth.login.$post`）を呼ぶ案 → React Router の form/action（progressive enhancement・SSR 時のリダイレクト + `Set-Cookie`）を失うため、in-process 呼び出しを採用

# Consequences

- レート制限・バリデーションが実際のログイン/サインアップ経路に適用されるようになった
- ログインCSRF（クロスサイトのフォームPOST）は `postAuthApi` の同一オリジン検証で遮断されるようになった
- composition root が API ハンドラーに一本化され、認証の横断的関心事の管理場所が1箇所になった
- action → API の in-process ホップが1段増える（Worker プロセス内呼び出しのためレイテンシ影響は軽微）
