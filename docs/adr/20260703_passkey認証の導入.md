# passkey(WebAuthn)認証の導入

Status: Proposed

Relevant PR:

-

# Context

認証はSupabase Authのメール+パスワードのみだった。フィッシング耐性が高くUXの良いpasskey(WebAuthn)を、ログインの選択肢として追加したい。

前提となる状況:

- Supabase Authが2026-05からpasskeyをネイティブ対応した(`auth.passkey.*`)。ただし**Beta / experimental**で、opt-in(`auth.experimental.passkey: true`)が必要かつAPI変更があり得る。
- `@supabase/supabase-js`は`^2.108.1`で、passkey要件(`>= 2.105.0`)を満たす。
- 認証情報(公開鍵クレデンシャル)はパスワード同様Supabase側に保存されるため、アプリのD1に新テーブルは不要。`active_users`は従来どおり`authentication_id`で紐づく。
- passkeyが使えるかは環境依存: 本番はSupabase、ローカル/E2Eは**supabase CLI**の対応可否による。結合テストで使う**supa-emu**は現状passkey未対応。

# Decision

Supabaseネイティブのpasskeyに乗り、**メール+パスワードを主のまま残した「選べる追加ログイン手段」**として導入する。ログイン画面に「パスキーでログイン」を併記するのみで、**登録を促す案内(バナー等)は出さない**。

## 実装方針

- 現アプリは「Supabase呼び出しはサーバー側に集約、セッションはhttpOnly Cookie」構造。passkeyのceremonyはブラウザの`navigator.credentials`を必ず経由するため、**低レベルの`auth.passkey.start*/verify*`をサーバーで実行し、ブラウザは`@simplewebauthn/browser`でWebAuthn ceremonyのみ担当**する。既存のpasswordフローと同じ`client → Hono API → domain(AuthUseCase) → SupabaseAuthRepository → Supabase`の一本道に載せる。
- Supabase呼び出しは`SupabaseAuthRepository`に隔離し、Betaの破壊的変更の影響範囲をそこに閉じ込める。

## Reason

- 自前WebAuthn実装(`@simplewebauthn/server` + D1クレデンシャルストア + セッション発行)に比べ、実装・保守コストが小さく、認証情報の保管をSupabaseに委ねられる。
- passwordを残す併存方式なら、Beta由来の不具合が起きても既存ログインは無影響。

## フラグとテスト戦略

- **フィーチャーフラグ`PASSKEY_ENABLED`**で機能全体をゲートする(`'true'`のときのみ有効)。passkey非対応の環境は既定OFF、対応環境でON。OFF時はpasskeyのルート/UIを出さず、passwordフローは不変。
- ドメイン/APIのテストは`AuthRepository`ポートのモックで完結させ、実Supabaseに依存させない。passkeyハンドラは薄い委譲層で、結合テストが使うsupa-emuがpasskey未対応の間は網羅できないためカバレッジ除外とする。
- E2Eは実際のsupabase CLIに対して実行する。CLIがpasskeyに対応するまでfixmeとし、対応後にフラグON + `config.toml`へ`[auth.passkey]`/`[auth.webauthn]`追加だけで実結線へ切り替えられる状態を保つ。

# Consequences

- passkey対応環境では、パスワードに加えパスキーでログインできる。登録の導線(トリガー)は案内バナーではなく、別途アカウント設定等に配置する。
- Beta / GA後に「推奨手段への格上げ」や「完全passwordless化」を別途検討できる余地を残す。
- 本番でpasskeyを使うにはSupabaseダッシュボードでの有効化と、RP ID / RP origins(本番ドメイン)の設定が必要。
