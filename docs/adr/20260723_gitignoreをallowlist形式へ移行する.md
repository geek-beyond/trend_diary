# .gitignoreをallowlist形式へ移行する

Status: Accepted

Relevant PR:

# Context

#1034 のフラット化で、ルートと `application/` の2枚の `.gitignore` を「統合のみ・内容無変更」で1枚にまとめた。中身は精査していなかったため、pnpm 運用の本リポジトリに無関係なテンプレート由来のエントリ（Yarn 系、pnpm 以外のパッケージマネージャのログ、JetBrains 系）が残っていた。

加えて現状は「無視するものを列挙する」ブラックリスト形式のため、テンプレートや他ツール由来のエントリが混入しても気づきにくく、追跡対象がファイル一覧から暗黙的にしか把握できない。

なお本リポジトリには、生成物を無視するための per-workspace の `.gitignore`（`apps/cron/.gitignore` / `apps/web/.gitignore` / `e2e/.gitignore` / `packages/datastore/.gitignore` / `supabase/.gitignore`）が既に存在する。これらは `.wrangler` / `build/` / `test-results/` / `*.db` 等、各 workspace 固有の生成物のみを対象にした簡潔なブラックリストであり、テンプレート由来の無関係なエントリという問題は見られない。

# Decision

## 不要エントリの削除

ルートの `.gitignore` から、pnpm 運用に無関係なエントリを削除する。

- Yarn 系（`.yarn/` / `!.yarn/releases`）: `package.json` の `packageManager` は pnpm であり Yarn は使わない
- 他パッケージマネージャのログ（`npm-debug.log*` / `yarn-debug.log*` / `yarn-error.log*` / `lerna-debug.log*`）: pnpm 以外は使わない。なお `*.log` の包括パターンで既にカバーされており、実体としての挙動差分は無い
- JetBrains 系（`.idea/workspace.xml` / `.idea/usage.statistics.xml` / `.idea/shelf`）: `.vscode/*` を明示的に管理下に置いており VSCode が前提。リポジトリ内に JetBrains 系ツールへの言及は無く、利用実態が確認できないため削除する。個々人のエディタ設定はプロジェクトの `.gitignore` ではなく `core.excludesFile`（グローバル gitignore）で管理する

## allowlist形式への移行（ルート1枚・トップレベルのみ許可）

ルートの `.gitignore` を `/*` で全無視し、ルート直下のみ `!` で追跡対象を列挙する allowlist 形式へ変更する。

- ディレクトリ（`apps/` `packages/` `docs/` 等）は `!/apps/` のようにトップレベルの1行を許可するだけでよい。gitignoreの仕様上、親ディレクトリ自体が無視から除外（`!`）されていれば、それ以降 git はそのディレクトリ配下を通常どおり走査するため、配下の新規ファイルは追加の `!` なしで追跡対象になる（「途中ディレクトリを1段ずつ `!` で開ける必要がある」のは、ディレクトリ内の一部ファイルだけを許可したい場合のみで、本リポジトリのように配下を丸ごと追跡したい場合には不要）
- per-workspace の生成物除外（`apps/web/.gitignore` 等）は、既に対象が生成物のみに絞られた簡潔なブラックリストであり、テンプレート由来の無関係なエントリという今回の問題を抱えていない。これを allowlist 化すると、各 workspace の `src/` 配下を丸ごと1行ずつ許可し直す必要が生じ、複雑さが増すだけでメリットが無いため、現状のブラックリスト形式のまま変更しない
- `.vscode/*` は個人設定を含みうるため、ディレクトリごと許可した上で、直下を再度無視し共有設定ファイルのみを許可する既存の入れ子構造（`.vscode/*` → `!.vscode/settings.json` 等）を維持する

検証: 移行前後で `git ls-files`（追跡対象）に差分が無いこと、および代表的な生成物パス（`node_modules/` `.env` `*.log` `coverage/` 等）と、既存の追跡対象ディレクトリ配下に新規ファイルを追加した場合の `git check-ignore` 判定が変わらないことを確認した。挙動が変わったのは今回削除した Yarn 系エントリ（`.yarn/` がネストした階層に出現した場合のみ）で、pnpm 運用では実際には発生しない想定内の差分である。

## 追加漏れのリスクと対応

allowlist 形式は「新規ファイルが無視されて気づかない」リスクを伴うが、上記の設計（配下を丸ごと許可する粒度）により、そのリスクは実質的に**リポジトリ直下に新しいファイル・ディレクトリを追加した場合のみ**に限定される（既存の追跡対象ディレクトリ配下の新規ファイルは通常どおり追跡される）。

CI での自動検知は、無視されたファイルはそもそも commit / push されないため、CI 上のチェックアウトには現れず効果が薄いと判断し採用しない。代わりに `.gitignore` 冒頭にコメントで運用ルール（ルート直下追加時は `!` の追記が必要）を明記し、レビュー時に気づける状態にする。

# Reason

- pnpm 運用に無関係なエントリを放置すると、今後もテンプレート由来のボイラープレートが無自覚に混入し続けるため、まず現状の棚卸しで削除する
- allowlist 形式はトップレベルの許可のみで足り、モノレポの深いネストに対しても「追跡対象を明示する」という目的を実用的なコストで達成できるため、ルート1枚のまま採用する
- per-workspace の `.gitignore` は既に精度の高いブラックリストとして機能しており、allowlist化する必然性が無いため、現状維持とする

検討した代替案:

- 各 workspace ごとに allowlist を分割する: モノレポのディレクトリ構成を `.gitignore` 側にも重複して表現することになり、構成変更のたびに複数ファイルを追随させる必要が生じるため見送った
- allowlist への移行自体を見送る: ブラックリスト形式のままだと、今回のようなテンプレート由来の無関係なエントリの混入に将来も気づきにくいままであるため、移行のコスト（トップレベル1行の追加のみ）に対してメリットが上回ると判断し採用した

# Consequences

- ルート直下に新しいファイル・ディレクトリを追加する際は、`.gitignore` に `!` を追記しない限り追跡されない。追記漏れは `git status`（無視されたファイルは表示されない）では気づけないため、追加時は `.gitignore` の更新を意識する必要がある
- 既存の追跡対象ディレクトリ（`apps/` `packages/` 等）配下でのファイル追加は、これまでどおり `.gitignore` の変更なしに追跡される
- Yarn / 他パッケージマネージャ / JetBrains 系のツールを新たに使う場合、必要なエントリを都度 `.gitignore` へ追加する（個人のエディタ設定は `core.excludesFile` を優先する）
